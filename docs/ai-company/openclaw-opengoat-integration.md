## 1. 現行アーキテクチャの課題

### 1.1 tool-integration-guide の前提

```
Human → Claude(Orchestrator) → AI-CFO / AI-CMO / AI-CTO / AI-COO
```

**この前提は合わない。** 理由:

| 前提 | 現実 |
|------|------|
| 機能別リーダー (CFO, CMO, CTO, COO) | プロジェクト別リーダーが必要 |
| 1つの事業を分業 | 複数プロジェクトを1人で並行運営 |
| CxOが常駐 | プロジェクトごとに必要なときに起動 |

### 1.2 現実の運営構造

```
夏元（1人）
├── AIカンパニー ← 研究 + 実行
├── VibeRush    ← devOps運用中
├── ZENT
└── マイクロビジネス ← 勉強会・超バイブコーディング
```

1人が複数プロジェクトを跨いで動く。必要なのは**機能別の参謀**ではなく、**プロジェクト別のAIリーダー**と、それらを**横断的に監視・構築するインフラ**。

## 2. OpenClaw と OpenGoat の関係

```
┌─────────────────────────────────────────────┐
│  OpenGoat（組織構築層）                       │
│                                              │
│  「どんなエージェントを、どう組むか」を設計する  │
│  ・エージェント群の構造定義                    │
│  ・プロジェクトリーダーの役割設計              │
│  ・チーム編成・再編成                         │
│  ・Agent-Teamの構成管理                      │
│                                              │
│  比喩: 建築家（組織の図面を引く）              │
└──────────────────┬──────────────────────────┘
                   │ 設計した構造の上で
                   ▼
┌─────────────────────────────────────────────┐
│  OpenClaw（ループ監視層）                      │
│                                              │
│  「設計した組織が実際に回っているか」を監視する  │
│  ・Close The Loop — ループが閉じているか      │
│  ・意思決定の追跡・固定                       │
│  ・フィードバックの還元確認                    │
│  ・ループ断絶の検知・アラート                  │
│                                              │
│  比喩: 現場監督（建てた建物が機能しているか）    │
└─────────────────────────────────────────────┘
```

**OpenGoatが組織を設計し、OpenClawがその組織のループを監視する。**

OpenGoatはOpenClawの上位レイヤーではなく、**同じ基盤（Close The Loop思想）の上に立つ2つの異なる機能**:
- OpenGoat = 構造の**設計**（静的）
- OpenClaw = 構造の**監視**（動的）

### 3.2 データフロー

```
意思決定の流れ（上→下）:
Human → Claude → OpenGoat(誰がやるか) → Project Leader → Agent-Team → 実行

ループ監視の流れ（下→上）:
実行結果 → OpenClaw(閉じたか？) → 断絶なら → Slack #ai-alerts → Human

組織再編の流れ（横）:
OpenClaw(このループ閉じない) → OpenGoat(構造を再設計) → 新しいAgent構成
```

## 4. OpenGoatの具体的な機能

### 4.1 エージェント構造定義

OpenGoatは各プロジェクトの「CLAUDE.md的な設計書」を生成・管理する。

**プロジェクトリーダー定義テンプレート:**

```yaml
# OpenGoat Project Leader Definition
project: ai-company
leader_name: AI-Company Leader
mission: AIカンパニーの研究推進とLoopDesk運用

responsibilities:
  - 研究ノートの管理と論文ドラフト支援
  - LoopDesk P2(5Gap変換)のAgent-Team編成
  - フィードバックの構造化と研究還元

agent_team:
  - role: reader
    task: 研究ノートを読み、要点を抽出
  - role: converter
    task: 5Gap変換ルールを適用し教材ドラフト生成
  - role: reviewer
    task: 整合性チェック、抜け漏れ確認

loops_to_close:  # ← OpenClawが監視する対象
  - name: research-to-teaching
    steps: [固定ポイント宣言, 5Gap変換, 教材提供, フィードバック還元]
    frequency: monthly
  - name: hypothesis-to-paper
    steps: [仮説, 実験, 結果分析, 論文更新]
    frequency: quarterly

authority_level: L2  # AIが実行、人間に報告
```

### 4.2 チーム再編成

OpenClawが「このループ閉じていない」と検知したとき、OpenGoatが構造を再設計する。

```
OpenClaw: 「VibeRushのLoop "課題→改善→デプロイ→効果測定" が2週間断絶」

  ↓ アラート

OpenGoat: 分析
  原因: VibeRush Leaderに効果測定の能力が定義されていない
  対策: Agent-Team に metrics-analyst ロールを追加

  ↓ 再編成

新構成:
  VibeRush Leader
  ├── developer (既存)
  ├── deployer (既存)
  └── metrics-analyst (NEW) ← 効果測定を担当
```

### 4.3 OpenGoat × OpenClaw の協調サイクル

```
OpenGoat                           OpenClaw
  │                                   │
  │ 1. 組織構造を設計                  │
  │──────────────────────────────────→│
  │                                   │ 2. 設計された構造で
  │                                   │    ループを監視開始
  │                                   │
  │                                   │ 3. ループ断絶を検知
  │←──────────────────────────────────│
  │ 4. 構造を再設計                    │
  │──────────────────────────────────→│
  │                                   │ 5. 新構造で監視再開
  │                                   │
  (このサイクル自体がClose The Loop)
```

```

  #ai-ops           ← エージェント稼働通知
  #ai-decisions     ← 意思決定ログ

変更:
  #ai-alerts        ← OpenClawからのループ断絶アラート
                       例: 🔴 AIカンパニー: フィードバック未還元 14日

  #loop-status      ← OpenClawからの定期ループステータス
```

## 6. LoopDeskとの統合

LoopDeskは、このアーキテクチャの中で以下のように位置づけられる:

```
OpenGoat: LoopDesk用のAgent-Team構成を定義
  │
  │ defines
  ▼
Agent-Team: LoopDesk P2 (5Gap変換) を実行
  │  Agent A(読解) → Agent B(変換) → Agent C(品質)
  │
  │ executes
  ▼
LoopDesk: 研究→教材→勉強会→フィードバック→研究
  │
  │ monitored by
  ▼
OpenClaw: LoopDeskのループが閉じているか監視
  │
  │ if broken → alert → OpenGoat reconfigures
  ▼
再設計サイクル
```

**LoopDeskは独立した「部門」ではなく、OpenGoat + OpenClaw + Agent-Team が協調して動作する「プロセス」。**

## 7. 1人運営における実装の現実

### 7.1 最小構成

1人で全部を同時に動かすのは非現実的。以下の優先順位で段階導入:

| 順序 | 導入するもの | 最小実装 |
|------|------------|---------|
| 1 | OpenClaw (監視) | ループ定義をYAMLで書く。手動でステータス確認 |
| 2 | プロジェクトLeader | CLAUDE.mdにリーダー定義を書く。Claude Codeセッションで使う |
| 3 | LoopDesk (変換) | 固定ポイント宣言1件 + 手動5Gap変換 |
| 4 | OpenGoat (設計) | Agent-Team構成をYAMLで定義。必要時にClaude Codeで再設計 |
| 5 | 自動化 | n8n / Slack連携で定期チェック |

### 7.2 「1人」だからこそのOpenGoat

1人で複数プロジェクトを回す最大の問題: **自分の頭の中にしか組織構造がない。**

OpenGoatで構造を明文化することで:
- Claude Codeセッションのたびに文脈を説明し直す必要がなくなる
- プロジェクトを切り替えても、AIリーダーが引き継ぐ
- 「このプロジェクトで今何をすべきか」をAIが把握できる

### 7.3 「1人」だからこそのOpenClaw

1人で複数プロジェクトを回す最大のリスク: **あるプロジェクトのループが気づかず切れる。**

OpenClawで全プロジェクトのループを可視化することで:
- 放置されているプロジェクトが一目で分かる
- 「今週VibeRushに何もしていない」が自動検知される
- Close The Loop を仕組みで担保する
