## 1. 位置づけ

### 1.1 二重の性格

AIカンパニーは**二重の性格**を持つ：

- **ビジョンとしてのAIカンパニー**: AIエージェント集団が意思決定の生成・評価・固定・学習を担い、人間は法的責任と最終拒否権のみを保持する組織形態
- **現実としてのAIカンパニー**: スプリントジャパン社内の1プロジェクト。VibeRush、ZENT、マイクロビジネスと同列

```
スプリントジャパン（会社）
├── AIカンパニー        ← 本プロジェクト
├── VibeRush            ← devOps運用中
├── ZENT
└── マイクロビジネス
    ├── 勉強会運営
    └── 超バイブコーディング講座
```

この「ビジョン ≠ 現在の運営単位」というギャップを認識した上で設計する。

## 3. ループデスク (LoopDesk) — 研究と教育の分離設計

### 3.1 なぜ「ループデスク」か

**名称の根拠:**

AIカンパニーの理論では、価値創出のポイントは「意思決定が不可逆的に固定される瞬間」にある（AI署名概念）。
研究→教育の変換は、まさに**知見の固定化 (Knowledge Fixation)** である。
流動的で探索的な研究知見を、他者に伝達可能な形に「固定」する行為。

同時に、勉強会の実践フィードバックが研究に戻ることで**ループが閉じる**。
OpenClawの「Close The Loop」と同じ原理が、知識の領域で働く。

| 候補 | 不採用理由 |
|------|-----------|
| ループデスク | 官僚的。一方向（研究→教育）のニュアンスが強い |
| ナレッジブリッジ | 橋は静的。ループの動的循環を表現できない |
| KT (Knowledge Transfer) | 汎用的すぎる。AIカンパニー固有の哲学が反映されない |
| **ループデスク** | **採用。Close The Loopと直結。「デスク」=軽量なサービス窓口** |

**「デスク」という語の意味:**
重い「部門」ではなく、軽量な「窓口」。
小規模組織でも運用可能な、プロトコル駆動のインターフェースであることを名称で示す。

### 3.2 問題の構造分析

**「研究内容がそのまま勉強会に使えない」の5つのギャップ:**

```
研究 (AIカンパニー)                   勉強会 (マイクロビジネス)
─────────────────                    ─────────────────────
理論的・探索的         ← 抽象度Gap →   「持ち帰って使える」実践
専門知識を前提         ← 前提知識Gap →  初学者〜中級者が対象
論文・実験ログ         ← 形式Gap →     スライド・ワークショップ
AIカンパニー所属       ← 組織Gap →     マイクロビジネス所属
未完成・進行中         ← 完成度Gap →   完成したコンテンツが必要
```

この5つのギャップを**同時に**解決するのがループデスクの役割。

### 3.3 3層モデル

```
┌─────────────────────────────────────────────────────┐
│  Layer 1: 研究（Research）                            │
│  ・論文、実験、仮説検証                                 │
│  ・アウトプット: 研究ノート、論文ドラフト                 │
│  ・所属: AIカンパニー                                   │
│  ・公開レベル: 限定的（IP保護）                          │
│                                                       │
│  ★ 固定ポイント宣言                                    │
│    研究者が「この知見は固定可能」と判断した時点で          │
│    ループデスクに変換リクエストを出す                     │
└──────────────┬──────────────────────────────────────┘
               │
      ┌────────▼────────┐
      │  ループデスク      │ ← 変換エンジン
      │  (LoopDesk)       │
      │                   │
      │ 入力: 固定可能な知見│
      │ 処理: 5Gap変換     │
      │ 出力: 教材パッケージ│
      └────────┬────────┘
               │
┌──────────────▼──────────────────────────────────────┐
│  Layer 2: 教材（Teaching Materials）                   │
│  ・体系化されたコンテンツ                               │
│  ・アウトプット: スライド、ワークシート、動画台本          │
│  ・所属: AIカンパニー（知的資産として保持）               │
│  ・公開レベル: マイクロビジネスに提供可                   │
└──────────────┬──────────────────────────────────────┘
               │ 提供プロトコル
┌──────────────▼──────────────────────────────────────┐
│  Layer 3: 勉強会・講座（Delivery）                      │
│  ・参加者向けの実施                                     │
│  ・所属: マイクロビジネス                               │
│                                                       │
│  ★ フィードバック固定                                  │
│    参加者の反応・質問・気づきを構造化して記録             │
│    ループデスク経由で Layer 1 に還元                    │
└──────────────┬──────────────────────────────────────┘
               │
      ┌────────▼────────┐
      │  ループデスク      │ ← 還元エンジン
      │  (フィードバック)  │
      │                   │
      │ 入力: 実践の記録   │
      │ 処理: 研究課題化   │
      │ 出力: 研究テーマ   │
      └────────┬────────┘
               │
               ▼
         Layer 1 に戻る
         (ループ完成)
```

### 3.4 ループデスクの運用設計

#### 3.4.1 「部門」ではなく「プロトコル」として運用する

小規模組織にとって、専任部門は現実的でない。
ループデスクは**3つのプロトコル（手順書）と2つのテンプレート**で運用する。

**プロトコル:**

| # | プロトコル名 | トリガー | 実行者 | 所要時間 |
|---|------------|---------|--------|---------|
| P1 | **固定ポイント宣言** | 研究者が「この知見は伝達可能」と判断 | 研究者本人 | 15分 |
| P2 | **5Gap変換** | P1が発行された時 | Agent-Team（AI支援） | 1-2時間 |
| P3 | **フィードバック還元** | 勉強会終了後 | 勉強会運営者 | 30分 |

**テンプレート:**

**T1: 固定ポイント宣言シート**
```
────────────────────────────────────
固定ポイント宣言
────────────────────────────────────
日付:
研究テーマ:
固定する知見（1文で）:
確信度: □高（実験済み） □中（仮説あり） □低（直感）
想定する勉強会トピック:
IP注意事項: □特許出願前のため非公開部分あり □制限なし
────────────────────────────────────
```

**T2: フィードバック還元シート**
```
────────────────────────────────────
フィードバック還元
────────────────────────────────────
勉強会日付:
使用した教材:
参加者数:
────────────────────────────────────
反応:
  ・最も関心が高かったトピック:
  ・理解が難しかったポイント:
  ・予想外の質問:
  ・「これを知りたい」という要望:
────────────────────────────────────
研究へのフィードバック:
  ・検証が必要な仮説:
  ・新しい研究テーマ候補:
  ・理論の修正が必要な箇所:
────────────────────────────────────
```

#### 3.4.2 5Gap変換の具体的手順

各ギャップに対する変換ルール:

| Gap | 研究側 | 変換操作 | 教材側 |
|-----|-------|---------|-------|
| **抽象度** | 理論命題 | 具体例を3つ付ける。「たとえば」から始まる文を追加 | 事例ベースの説明 |
| **前提知識** | 専門用語使用 | 用語を日常語に置換。専門用語は補足欄へ | 平易な言葉 |
| **形式** | 論文形式 | 下表の変換マトリクスに従う | ワークショップ形式 |
| **組織** | AIカンパニー資産 | 教材パッケージとしてマイクロビジネスに提供 | マイクロビジネスが実施 |
| **完成度** | 進行中 | 「固定ポイント宣言」済みの知見のみ変換対象 | 完成教材 |

**形式変換マトリクス:**

| 研究アウトプット | 変換先フォーマット | 構成 | 例 |
|-----------------|------------------|------|-----|
| 理論論文 | 概念解説スライド（15分） | 問い→仮説→証拠→結論→議論 | CEO機能分解 → 「AIに任せていい判断、ダメな判断」 |
| 実験ログ | ハンズオンWS（60分） | 背景(5分)→デモ(15分)→実習(30分)→振り返り(10分) | Agent-Team実験 → 「Claude Codeでチームを組んでみよう」 |
| 技術検証 | ライブデモ + 手順書 | 目的→環境構築→実演→トラブルシューティング | OpenClaw → 「生成AIを設定して動かすまで」 |
| ビジネスモデル仮説 | ケーススタディ（30分） | 状況→課題→仮説→検証→学び→あなたなら？ | Close The Loop → 「フィードバックで事業を回すには」 |

#### 3.4.3 Agent-Teamとの連携

**5Gap変換はClaude Code Agent-Teamに委任できる。**

Agent-Teamがループデスクの変換エンジンを担当する運用フロー:

```
研究者
  │ 固定ポイント宣言 (T1)
  ▼
Agent-Team
  │ 1. 研究ノートを読み込み
  │ 2. 5Gap変換を実行
  │ 3. 教材ドラフトを生成
  │ 4. 研究者がレビュー・修正
  ▼
教材パッケージ完成
  │ マイクロビジネスに提供
  ▼
勉強会実施
  │ フィードバック還元シート (T2)
  ▼
Agent-Team
  │ フィードバックを構造化
  │ 研究テーマ候補として整理
  ▼
研究者にフィードバック還元
```

これにより:
- **研究者の負荷**: 固定ポイント宣言（15分）+ レビュー（30分）のみ
- **変換作業の大部分**: Agent-Teamが処理
- **勉強会運営者の負荷**: フィードバックシート記入（30分）のみ

#### 3.4.4 ループが回る条件

ループデスクが形骸化しないための**3つの最低条件**:

1. **月1回以上の固定ポイント宣言**: 研究が動いている証拠。なければ研究活動自体を見直す
2. **勉強会ごとのフィードバック還元**: 1回でも欠けるとループが切れる。T2を必須化
3. **四半期ごとの棚卸し**: 変換済み教材の利用実績と、還元フィードバックの研究反映率を確認

### 3.5 課題解決のまとめ

**Before（現状）:**
```
研究 ──（そのまま）──→ 勉強会で使おうとする ──→ 使えない
                                                    │
                                              フィードバックなし
                                              研究は孤立
```

**After（ループデスク導入後）:**
```
研究 ──→ 固定ポイント宣言 ──→ ループデスク ──→ 教材 ──→ 勉強会
  ▲                            (5Gap変換)                  │
  │                            Agent-Team支援               │
  │                                                         │
  └──── ループデスク ◀──── フィードバック還元 ◀──────────────┘
        (研究課題化)
```

**解決される問題:**

| 問題 | 解決策 | 担保する仕組み |
|------|-------|---------------|
| 研究がそのまま教材にならない | 5Gap変換プロセスで明示的に変換 | 変換マトリクス + Agent-Team |
| 変換の負荷が研究者に集中 | Agent-Teamに委任 | 研究者は宣言+レビューのみ |
| 勉強会の知見が研究に戻らない | フィードバック還元プロトコル | T2シート必須化 |
| 組織境界で情報が断絶 | ループデスクが公式インターフェース | 提供プロトコル |
| IP保護と公開のバランス | 固定ポイント宣言にIP注意欄 | 特許出願前は非公開指定可 |

## 4. プロジェクト間接続

### 4.1 AIカンパニー → マイクロビジネス

```
AIカンパニー                    マイクロビジネス
┌──────────┐    教材提供      ┌──────────────┐
│ LoopDesk │ ──────────────→ │ 勉強会運営    │
```

## 6. 設計原則

1. **研究は研究のまま蓄積する** — 教材化のプレッシャーで研究の質を下げない
2. **変換は明示的なプロセスとする** — 研究→教材の変換はループデスクの5Gap変換で行う。「なんとなく教材化」は禁止
3. **組織境界を尊重する** — AIカンパニーとマイクロビジネスは別プロジェクト。接続はインターフェースを通じて行う
4. **Close The Loopを実践する** — 勉強会のフィードバックを研究に戻すサイクルを設計に組み込む
5. **段階的に移行する** — ビジョン（AIカンパニーが最上位）への移行は、実績の積み上げで自然に起こるようにする
