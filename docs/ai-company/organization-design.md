## 1. 位置づけ

### 1.1 二重の性格

AIカンパニーは**二重の性格**を持つ：

- **ビジョンとしてのAIカンパニー**: AIエージェント集団が意思決定の生成・評価・固定・学習を担い、人間は法的責任と最終拒否権のみを保持する組織形態
- **現実としてのAIカンパニー**: スプリントジャパン社内の1プロジェクト。VibeRush、ZENT、マイクロビジネスと同列

```
スプリントジャパン（会社）
├── AIカンパニー        ← 本プロジェクト
├── VibeRush            ← devOps運用中
├── ZENT
└── マイクロビジネス
    ├── 勉強会運営
    └── 超バイブコーディング講座
```

この「ビジョン ≠ 現在の運営単位」というギャップを認識した上で設計する。

## 2. AIカンパニーの内部構造

### 2.1 2つのチーム

| チーム | ミッション | 役割 |
|--------|-----------|------|
| **OpenClaw** | Close The Loopの推進 | フィードバックループを閉じ、意思決定を不可逆的に固定するプロセスを構築 |
| **Claude Code Agent-Team** | 業務遂行チームの再組成 | AIエージェントチームによる実務オペレーションの実行 |

### 2.2 構造図

```
AIカンパニー（プロジェクト）
│
├── 【研究部門】 ─── 理論・実験・知見の蓄積
│   ├── AI組織論研究（ai_company_paper）
│   ├── CEO機能分解モデル
│   ├── AI署名概念
│   └── 意思決定インフラ実験
│
├── 【実行部門】
│   ├── OpenClaw チーム
│   │   ├── Close The Loop プロセス設計
│   │   ├── 生成AI設定・運用
│   │   └── フィードバックループ実装
│   │
│   └── Claude Code Agent-Team
│       ├── エージェント構成設計
│       ├── タスク分解・委譲フレームワーク
│       └── 業務遂行パイプライン
│
└── 【教育接続部門】 ─── 研究→教材への変換レイヤー

    ├── 勉強会向けコンテンツ設計
    └── マイクロビジネスへの提供インターフェース
```

## 3. 研究と教育の分離設計

### 3.1 問題

**研究内容がそのまま勉強会に使えない。**

理由：
- 研究は理論的・探索的であり、完成度や再現性が保証されていない
- 勉強会は参加者が「持ち帰って使える」形式が求められる
- 研究のアウトプット（論文、実験ログ）と教育のアウトプット（カリキュラム、ワークシート）はフォーマットが異なる
- 研究はAIカンパニーに属し、勉強会はマイクロビジネスに属する（組織境界がある）

### 3.2 解決: 3層モデル

```
┌─────────────────────────────────────────────────┐
│  Layer 1: 研究（Research）                        │
│  ・論文、実験、仮説検証                             │
│  ・アウトプット: 研究ノート、論文ドラフト             │
│  ・所属: AIカンパニー                               │
│  ・公開レベル: 限定的（IP保護）                      │
└──────────────┬──────────────────────────────────┘
               │ 変換プロセス（教育接続部門が担当）
               │ ・専門用語の平易化
               │ ・理論→実践演習への翻訳
               │ ・ケーススタディ化
               ▼
┌─────────────────────────────────────────────────┐
│  Layer 2: 教材（Teaching Materials）               │
│  ・体系化されたコンテンツ                           │
│  ・アウトプット: スライド、ワークシート、動画台本     │
│  ・所属: AIカンパニー（資産として保持）              │
│  ・公開レベル: 社内共有可                           │
└──────────────┬──────────────────────────────────┘
               │ 提供インターフェース
               │ ・マイクロビジネスとの連携プロトコル
               │ ・勉強会フォーマットへの適合
               ▼
┌─────────────────────────────────────────────────┐
│  Layer 3: 勉強会・講座（Delivery）                  │
│  ・参加者向けの実施                                 │
│  ・アウトプット: 勉強会実績、参加者フィードバック     │
│  ・所属: マイクロビジネス                           │
│  ・公開レベル: 外部公開                             │
└─────────────────────────────────────────────────┘
```

### 3.3 変換ルール

| 研究のアウトプット | 変換後の教材形式 | 例 |
|-------------------|-----------------|-----|
| 理論論文 | 概念解説スライド（10分） | CEO機能分解 → 「AIに任せられる仕事、任せられない仕事」 |
| 実験ログ | ハンズオンワークショップ | Agent-Team実験 → 「Claude Codeでチームを作ろう」 |
| 技術検証 | デモ + チュートリアル | OpenClaw設定 → 「生成AIの設定ガイド」 |
| ビジネスモデル仮説 | ケーススタディ | Close The Loop → 「フィードバックで事業を回す方法」 |

### 3.4 フィードバックループ

勉強会での参加者の反応・質問は、Layer 3 → Layer 1 へフィードバックとして戻す。
これ自体が**Close The Loop**の実践になる。

## 4. プロジェクト間接続

### 4.1 AIカンパニー → マイクロビジネス

```
AIカンパニー                    マイクロビジネス
┌──────────┐    教材提供      ┌──────────────┐
│ 教育接続  │ ──────────────→ │ 勉強会運営    │
│ 部門     │ ←────────────── │              │
└──────────┘  フィードバック   └──────────────┘
```

- AIカンパニーは教材を提供する**サプライヤー**
- マイクロビジネスは勉強会を実施する**デリバリー**
- 双方向のフィードバックで改善サイクルを回す

### 4.2 AIカンパニー → VibeRush

VibeRushはdevOpsで運用中。AIカンパニーのAgent-Teamの知見をVibeRushの運用改善に適用できる可能性がある。

### 4.3 全体マップ

```
                    ┌─────────────┐
                    │ AIカンパニー  │
                    │ (研究・実行)  │
                    └──┬───┬───┬──┘
            教材提供 │   │   │ 運用知見
                     │   │   │
          ┌──────────▼┐ │ ┌─▼──────────┐
          │マイクロ    │ │ │ VibeRush   │
          │ビジネス    │ │ │ (devOps)   │
          │・勉強会    │ │ └────────────┘
          │・超ﾊﾞｲﾌﾞ講座│ │

                        │
```

## 6. 設計原則

1. **研究は研究のまま蓄積する** — 教材化のプレッシャーで研究の質を下げない
2. **変換は明示的なプロセスとする** — 研究→教材の変換は「教育接続部門」が意図的に行う
3. **組織境界を尊重する** — AIカンパニーとマイクロビジネスは別プロジェクト。接続はインターフェースを通じて行う
4. **Close The Loopを実践する** — 勉強会のフィードバックを研究に戻すサイクルを設計に組み込む
5. **段階的に移行する** — ビジョン（AIカンパニーが最上位）への移行は、実績の積み上げで自然に起こるようにする
